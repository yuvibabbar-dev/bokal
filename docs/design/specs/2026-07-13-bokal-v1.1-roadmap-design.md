# Bokal v1.1 — Roadmap Program Design (M8–M10)

- **Date:** 2026-07-13
- **Status:** Approved (decisions locked with the founder) — ready for per-milestone planning
- **Source research:** [`docs/business/2026-07-13-feature-roadmap-research.md`](../../business/2026-07-13-feature-roadmap-research.md) (deep-research, 20 confirmed / 5 refuted claims)
- **Base:** `master` @ post-M7 (77 commits; 60 tests; tsc clean; E2E green both builds; published manifest has no `host_permissions`)

This program covers the post-M7 feature roadmap. It is decomposed into three milestones, each built through the repo's proven loop: plan → TDD execution → whole-branch review → merge. Each milestone ships independently and preserves every §7 invariant in [`docs/HANDOFF.md`](../../HANDOFF.md).

## Locked decisions (founder, 2026-07-13)

1. **Export-all host access → persistent `<all_urls>` opt-in.** Export/import across all domains needs broad host access. Bokal requests `<all_urls>` at runtime, inside the user's click gesture, the first time they use an all-domains action, and keeps it. **Install-time manifest is unchanged** (`optional_host_permissions` only) — so the store claim "no broad permissions at install" stays literally true. The threat model + privacy policy must be updated to describe this user-initiated persistent grant.
2. **Free/Pro split → roadmap recommendation.** Free: all CRUD, export/import-all, one-off automation-format exports (Playwright/Puppeteer JSON), protect/pin/block basics, auto-delete basics, DevTools panel, popup. Pro: named profiles (existing), storageState round-trip *suites* (save/apply named automation profiles), scheduled/advanced rule-sets. Core stays free to preserve Featured-badge eligibility.
3. **Build approach → autonomous, report per milestone.** Build M8 → M9 → M10 back-to-back; summarize after each; no mid-milestone approval gates.

## Invariants preserved across all three milestones (do not regress)

- No `tabs` permission; **no install-time `host_permissions`** (the `BOKAL_E2E` gate is test-only). New broad access is runtime-requested via `optional_host_permissions`.
- Free build ships **zero Pro code** — Pro surfaces load via dynamic `import()` behind the entitlement gate (verified as a separate chunk in CI-adjacent checks).
- Never log cookie values / passphrases — `lib/security/redaction.test.ts` scans all shipped source + `@bokal/ui-kit`.
- Cookie values render as **text nodes only** (never `dangerouslySetInnerHTML`).
- `chrome.storage` is the source of truth; theme pref stays `storage.local` (not `sync`).
- Every trust/permission claim in `docs/store/` + `docs/threat-model.md` stays literally true against the code.

## Bug fix (ships first, in M8): CHIPS partitioned cookies missing from the all-cookies view

`getAllCookies()` in `lib/cookies/read.ts` calls `chrome.cookies.getAll({})`, and the API returns **only unpartitioned cookies by default** — so the M7 "All cookies" scope silently omits partitioned (CHIPS) cookies. Fix: additionally query partitioned cookies and merge, deduping by `cookieId`. Verify the exact `getAll` partition argument against the live Chrome cookies API during TDD (the reference states methods default to unpartitioned; the partition query form must be confirmed, not assumed). Same latent gap should be checked for `getCookiesForUrl` when CHIPS is not toggled — but scope the fix to the all-view (site view already has the explicit partitioned path behind the toggle).

## M10 — Strategic bets (branch `feat/m10-autodelete`); Large effort

| Task | Feature | Free/Pro | Notes |
|---|---|---|---|

## Testing & workflow (every milestone)

- **TDD:** unit tests first for each lib change (Vitest + hand-mocked `chrome.cookies`); component tests where render logic is non-trivial (text-node/XSS locks).
- **Gates per milestone:** `pnpm -r test` green · `tsc --noEmit` clean · `build` + `zip` succeed · published manifest still has no `host_permissions` · redaction guard green · ProfilesPanel still a separate chunk · Playwright E2E green on both builds.
- **Review:** whole-branch review at milestone end → fix Critical/Important → merge `--no-ff` → delete branch.
- **Out of scope (unchanged):** payments/ExtPay wiring; account-bound store actions; session-switching (SessionBox-class) until CWS-review demand is validated; Firefox port.

## Open items to resolve during implementation (not blocking)

- Exact `chrome.cookies.getAll` partition-query form for the bug fix (verify against live API).
- storageState localStorage gap: `chrome.cookies` can't capture localStorage, so token-in-localStorage apps round-trip only partially — document in the export UI; do **not** add a content script for localStorage capture in v1.1 (permission-cost vs. trust).
- Popup vs. side-panel action-click behavior reconciliation (M9-T3).
